# Firmus Linux — AI Agent and Contributor Guidelines

## Core Principles

1. **Real builds only** — No placeholders, no simulations, no examples in build-critical files
2. **Zero assumptions** — Always verify package versions, configurations, and dependencies against current upstream data
3. **Minimalist code** — No unnecessary comments, no debug stubs, no dead code paths
4. **Consistent patterns** — Every file follows the same style as existing files of its type

## File Conventions

### PKGBUILD Files
```bash
# Maintainer must be listed at top
# Maintainer: Llucs <c307lucas@gmail.com>

pkgname=        # lowercase, hyphen-separated
pkgver=         # semver or upstream version
pkgrel=         # integer, starts at 1
pkgdesc=        # brief, lowercase, no period
arch=('x86_64' or 'any')
url=            # project URL
license=        # SPDX identifier
depends=()      # alphabetical
source=()       # URLs or inline functions
sha256sums=()   # SKIP only for generated content

package() {
  # Standard install section
  install -d "$pkgdir/usr/share/firmus/..."
  install -m 644 "$srcdir/..." "$pkgdir/..."
}
```

### Shell Scripts
- `set -euo pipefail` at top of every script
- Functions with `local` variables
- `main "$@"` pattern at bottom
- Color output using ANSI escape sequences
- `log_info`, `log_ok`, `log_warn`, `log_error` pattern for logging
- `check_root()` function when root required

### GitHub Actions Workflows
- Use `archlinux/archlinux:latest` container with `--privileged`
- Always run `pacman-key --init && pacman-key --populate archlinux` first
- `actions/checkout@v5` for checkout
- Use `softprops/action-gh-release@v3` for releases
- Use `peaceiris/actions-gh-pages@v4` for Pages deployment
- Use `actions/github-script@v7` for API calls

### Calamares Modules
- Module descriptor: `module.desc` in YAML
- Configuration: `<module-name>.conf` containing a `command:` field
- Commands run inside chroot via `{{{ ROOT }}}` variable
- Use `{{{ INSTALL_USER }}}` for target username

### Systemd Services
- `[Unit]` with `Description=` and `Documentation=`
- `[Service]` with `Type=oneshot` for single-run services
- `[Install]` with `WantedBy=`
- Services prefixed with `firmus-`

### Configuration Files
- Dracut: files in `/etc/dracut.conf.d/` prefixed `XX-firmus-NAME.conf`
- Systemd: drop-ins in `/etc/systemd/NAME.conf.d/firmus.conf`
- Pacman hooks: in `/etc/pacman.d/hooks/`
- Snapper: in `/etc/snapper/configs/root`

## Repository Structure Rules

```
firmuslinux/
├── .github/workflows/     # CI/CD — one workflow per concern
├── iso/                   # archiso profile — mirrors installed system
│   ├── profiledef.sh      # ISO metadata
│   ├── packages.x86_64    # Package list for live ISO
│   ├── pacman.conf        # Build-time pacman config
│   └── airootfs/          # Overlay for live system
├── packages/tcr/          # One directory per package
│   ├── tier1.csv          # Core packages (human review)
│   ├── tier2.csv          # Community packages (CI review)
│   └── tier3.csv          # AUR cache (automated)
├── calamares/modules/     # One directory per module
├── configs/               # System configs (mirrors airootfs)
├── scripts/               # Build and maintenance tools
├── branding/              # Theme assets
├── security/              # Policy and keys
└── testing/               # Test scripts and cases
```

## Workflow Rules

### ISO Build
- Weekly cron: `0 6 * * 0`
- Always use `archlinux/archlinux:latest` container
- Must run QEMU smoke test after build
- ISO published as GitHub Release with SHA256 checksum

### TCR Build
- Weekly cron: `0 8 * * 1`
- Build in isolated container
- Run `namcap` on all packages
- Sign packages with Firmus GPG key
- Deploy to `gh-pages` branch

### AUR Verification
- Daily cron: `0 10 * * *`
- Compare current PKGBUILD hashes against allowlist
- Open issue for each changed package

## Branch Strategy

- `main` — Production-ready state
- `gh-pages` — TCR repository (auto-deployed)
- Feature branches: `feat/<name>`
- Fix branches: `fix/<name>`

## Commit Messages

```
type: brief description

Optional body explaining motivation.
```

Types: `feat`, `fix`, `chore`, `docs`, `security`, `config`

## Versioning

- ISO: `YYYY.MM.DD` format (date-based rolling)
- Packages: semver (`MAJOR.MINOR.PATCH`)
- TCR tier files: no versioning (live)

## Security Rules

- Never hardcode GPG keys in scripts
- Never commit `.sig` files (generated by CI)
- Always use HTTPS for package sources
- Never use `SKIP` in PKGBUILD sha256sums for downloaded content
- All TCR packages must be signed
